# 工作日志与多端 Git 同步规范

本文件规定项目每次工作结束时的记录、提交、GitHub 推送和多端同步方式。目标是让本机、GitHub 和 sophgo13 服务器始终能够明确判断：完成了什么、验证到了什么程度、还有什么问题，以及三端是否处于同一提交。

## 1. 仓库与端点

- 本机主仓库：`D:\Personal\Code\parameter-importance`
- GitHub 远端：`https://github.com/Loong-C/parameter-importance.git`，远端名为 `origin`
- sophgo13 服务器仓库：`/home/sophgo13/cjl/parameter-importance`
- 服务器大盘：`/home/sophgo13/cjl/storage/parameter-importance`
- 连接、网络和大盘边界以本机忽略目录中的 `Agent/remote_access.md` 为准。

默认以本机仓库作为代码编辑端。不要在本机与服务器同时修改同一文件；如果工作确实从另一端开始，应先把该端形成的提交同步回来，再继续编辑。

## 2. 每次工作结束后的日志

日志放在仓库 `worklogs/`，使用中文 Markdown。文件名优先采用：

```text
YYYY-MM-DD-主题.md
```

同一天继续同一任务时，追加到原文件并标注北京时间；不同任务使用不同文件。每次记录至少包含：

1. 本次目标与明确范围；
2. 实际修改的代码、配置、文档和外部系统；
3. 关键产物的路径、固定 revision、大小或哈希；
4. 执行的检查、测试和实验，以及精确结果；
5. 遇到的问题、根因、采取的恢复措施；
6. 仍未解决的问题和风险；
7. 当前阶段、下一步动作和是否可以交给后续工作；
8. Git 与多端同步结果。

不要只写“测试通过”。应记录足以复核结论的信息，例如测试名称、用例数、退出状态、关键指标或报告路径。实验日志还应记录随机种子、配置文件、数据/模型 revision、checkpoint 和结果目录。

日志不得包含密码、私钥、令牌、Cookie、完整签名 URL、临时 CDN IP、认证头或其他秘密。模型、数据集、wheel、checkpoint 和大型运行产物不提交 Git；日志只记录其受控路径、大小、哈希和验证结论。

## 3. 修改前检查

开始工作前先确认当前分支、远端和工作树：

```powershell
git status -sb
git branch --show-current
git remote -v
git fetch origin
git rev-parse HEAD
git rev-parse origin/main
```

如果存在不属于本次工作的本机改动，必须保留并绕开，不能擅自覆盖、暂存或删除。若本机与 `origin/main` 已分叉，停止自动同步，先查明两个方向的提交；不得用 `git reset --hard`、强制推送或覆盖整个仓库来“解决”。

## 4. 本机提交与 GitHub 推送

完成实现和验证后，先更新工作日志，再检查实际差异：

```powershell
git status --short
git diff --check
git diff --stat
git diff
```

只显式暂存本次工作涉及的文件：

```powershell
git add -- <文件1> <文件2> <工作日志>
git diff --cached --check
git diff --cached
git commit -m "<准确、简短的提交说明>"
```

本项目只有在用户明确要求直接更新 `main` 时才直接推送 `main`；否则使用独立分支。推送前再次获取远端状态，并且只允许快进关系：

```powershell
git fetch origin
git merge --ff-only origin/main
git push origin main
```

如果 `git merge --ff-only` 或 `git push` 报告分叉、非快进或远端新提交，停止并检查，不得强推。网络 reset、timeout 或 empty reply 属于传输失败；保留本地提交，确认 `git ls-remote` 后重试，不要重复创建内容相同的提交。

推送后核对：

```powershell
git rev-parse HEAD
git ls-remote origin refs/heads/main
git status --short
```

本机 HEAD 与 GitHub `refs/heads/main` 必须完全相同，工作树应为空。

## 5. sophgo13 服务器的安全快进同步

服务器系统 DNS 和 GitHub 控制面可能不稳定，因此优先从已经完成推送的本机提交创建 Git bundle，经既有跳板连接传入服务器。不要复制整个仓库，不要覆盖 `.git`。

在本机创建并验证 bundle：

```powershell
$short = git rev-parse --short HEAD
git bundle create ".sync-$short.bundle" main
git bundle verify ".sync-$short.bundle"
scp ".sync-$short.bundle" "sophgo13-via-lab:/home/sophgo13/cjl/storage/parameter-importance/tmp/repo-sync-$short.bundle"
```

同步前，服务器仓库必须干净：

```powershell
ssh sophgo13-via-lab "cd /home/sophgo13/cjl/parameter-importance && git status --short && git rev-parse HEAD"
```

若服务器有未提交改动，停止同步并先判断改动归属。仓库干净时，在服务器执行：

```bash
cd /home/sophgo13/cjl/parameter-importance
git fetch /home/sophgo13/cjl/storage/parameter-importance/tmp/repo-sync-<short>.bundle main
git merge --ff-only FETCH_HEAD
git rev-parse HEAD
git status --short
```

确认服务器 HEAD 等于本机 HEAD 后，只删除这次同步使用的确切临时 bundle；不要使用通配符或递归清理。随后删除本机对应的确切临时 bundle。

`Agent/` 被 Git 忽略。若 `Agent/remote_access.md` 有修改，必须在不包含秘密的前提下单独复制到服务器同名路径，并比较两端 SHA-256；它不会随 GitHub 提交传播。

## 6. 从服务器或另一台本机产生修改时

如果修改最初发生在服务器或另一台本机：

1. 先确认该端工作树中只有本次改动；
2. 在来源端完成测试、日志和提交；
3. 将提交通过 GitHub 或经过验证的 bundle 送回本机主仓库；
4. 本机只做 `--ff-only` 合并；
5. 再从本机统一推送 GitHub 并同步其他端点。

不要用 SCP 单独覆盖受 Git 跟踪的代码文件来替代提交同步；紧急部署文件后，也必须尽快形成相同 Git 提交并核对内容哈希。

## 7. 完成判定

一次工作只有同时满足下列条件才算完成：

- 请求范围内的实现或处理已经结束；
- 相关检查、测试或实验已经运行并记录；
- 中文工作日志已更新；
- 本机工作树干净；
- 本机 HEAD 与 GitHub `refs/heads/main` 一致；
- sophgo13 仓库通过 `--ff-only` 到达同一 HEAD；
- 本机与服务器的工作日志内容一致；
- 临时 bundle 已按确切路径清理；
- 遗留问题已在日志和最终回复中明确说明。

推荐在最终核对时同时输出三个提交号和日志 SHA-256。三端不一致时不得宣称“已同步完成”。
