# S0.11 自动测试、端到端重放与交接文档

## 目的

把前述基础设施转化为可持续回归的测试体系，并证明新会话或其他执行者只依靠仓库、服务器资产 manifest 和运行说明就能离线重放单卡、四卡和恢复 smoke test。

## 当前依据

- 当前 `main` 没有现行 `src`、`tests`、运行入口或 Stage 0 报告生成器。
- 旧实验轻量归档可提供历史失败模式和报告字段参考，但不能作为现行测试通过证据。
- 本机没有 CUDA，必须明确测试分层，避免 GPU 测试在本机被错误计作通过。
- 后续 Stage 1 要在基础设施之上做数值正确性验证，必须有稳定 fixture 和回归报告。

## 前置条件

- G0–G8 的组件和证据已经形成，所有硬件、资产和容量子 gate 都是当前有效状态。
- 每个测试都能使用 G4 的配置、run ID 和 provenance。
- 测试输出遵守 G1 的存储边界。

## 实施步骤

### 1. 定义测试分层

1. 本机 CPU 层覆盖配置、schema、路径、manifest、run ID、seed 和纯函数。
2. 服务器 CPU 层覆盖 Linux 路径、离线资产元数据、日志和 checkpoint 结构。
3. 单 GPU 层覆盖真实 CUDA、BF16、模型和数据闭环。
4. 四 GPU 层覆盖 NCCL、DDP、分片、累积、`no_sync` 和恢复。
5. 故障层覆盖非法输入、损坏文件、单 rank 失败、日志失败和空间不足预检。
6. 重放层从空输出目录重建一次完整 Stage 0 smoke 流程。

### 2. 建立确定性 fixture

1. 创建极小模型、固定输入 token 和固定 labels。
2. 固定模型初始化摘要、样本 ID 和期望 loss/梯度摘要。
3. 将 fixture 控制在适合 Git 的体积内。
4. 记录生成器和数学语义，不存来源不明二进制。
5. fixture 版本变化时显式更新预期值和原因。
6. Pythia 14M 真实资产测试与极小 fixture 分开，避免大文件进入 Git。

### 3. 建立单元测试

1. 测试配置加载、覆盖、规范化和跨字段验证。
2. 测试 run ID 碰撞、attempt 和父子关系。
3. 测试 seed 域独立性和可重复性。
4. 测试路径只能落在允许根目录。
5. 测试资产 manifest 合法/非法/缺失/错误哈希分支。
6. 测试资产获取入口的中断续传、错误范围响应、缺 wheel/缺文件、错误哈希、锁竞争、幂等和原子发布分支。
7. 测试事件 schema、状态转换、canonical lineage、raw 分叉保留和敏感信息过滤。
8. 测试 checkpoint manifest、索引和损坏拒绝。
9. 测试容量估算、统一测量协议字段和启动前阻断条件。

### 4. 建立集成测试

1. 运行 CPU 极小闭环，验证配置到报告的全链路。
2. 运行单卡 FP32 确定性闭环。
3. 运行单卡 Pythia 14M BF16 闭环。
4. 运行四卡 DDP 等价闭环。
5. 运行梯度累积与 `no_sync` 闭环。
6. 运行单卡和四卡 checkpoint 恢复闭环。
7. 运行受控故障并检查状态、日志和无残留资源。

### 5. 生成统一测试报告

1. 每个测试记录 ID、层级、配置摘要、环境 ID、开始/结束时间和退出状态。
2. 记录断言数、通过/失败/跳过数量。
3. GPU 测试记录设备映射和健康摘要。
4. 数值测试记录容差、最大误差和最差张量。
5. 性能测试记录基线和相对变化。
6. 输出机器可读 JSON 和便于审阅的 Markdown 摘要。
7. 失败报告保留完整证据路径，不把失败覆盖成最终通过。

### 6. 编写运行说明

1. 说明本机与服务器职责和最小前置条件。
2. 说明如何识别推荐环境、资产 ID 和配置模板。
3. 说明如何选择单卡和四卡而不硬编码物理编号。
4. 说明 smoke、正式运行、恢复和故障诊断的流程。
5. 说明所有输出目录、状态文件和报告的位置。
6. 说明何时旧 gate 因环境漂移而失效。
7. 说明明确禁止的操作，包括绕过 manifest、使用 `.part`、修改 SSH 拓扑和系统级安装。

### 7. 做独立端到端重放

1. 由未参与实现的新会话或独立执行者阅读运行说明。
2. 创建新的唯一空输出目录开始，既不清理也不复用任何旧 run 状态。
3. 在受控网络阻断或连接审计条件下，使用锁定环境和完全离线资产运行 CPU/单卡 smoke，并保存零外连尝试证据。
4. 使用健康的四卡候选运行 DDP 与 `no_sync` smoke。
5. 从生成的 checkpoint 恢复并完成剩余步骤。
6. 只凭 run ID 定位配置、日志、checkpoint、结果和报告。
7. 对照预期证据清单，记录文档缺口并修订后重放。

### 8. 定义回归触发条件

1. 依赖锁、Python、驱动、内核、GPU 拓扑变化时重跑相应层。
2. 配置 schema、训练 step、loss reduction、日志或 checkpoint 改动时重跑相关数值层。
3. 模型/tokenizer/data manifest 变化时重跑资产和真实模型层。
4. 只改文档时运行链接、结构和一致性检查，不无意义占用 GPU。
5. 测试报告明确标注哪些 gate 是当前有效证据。

## 产出

- 分层自动测试套件和确定性 fixture；
- CPU、单卡、四卡、恢复和故障集成测试；
- 统一 JSON/Markdown 测试报告；
- 实验运行、恢复、故障诊断和证据定位说明；
- 独立端到端重放报告；
- 环境漂移与回归触发矩阵。

## 核验标准

- 本机 CPU、服务器 CPU、单 GPU、四 GPU 和故障测试的层级标记明确，跳过不被计作通过。
- 全部单元测试通过；报告给出用例数、断言/结果和退出状态。
- G5–G7 的数值、恢复和错误处理标准在集成测试中自动断言。
- 测试报告能从 run ID 追溯到 Git、配置、环境、设备、资产和 checkpoint。
- 独立执行者在不联网、不询问隐含步骤的情况下完成单卡、四卡和恢复重放。
- 重放生成的新 run ID 不覆盖原证据，关键数值落在既定容差内。
- 文档中的路径、链接、产物清单和禁止项与实际实现一致。
- 任何失败或跳过的硬 gate 测试都会使总报告失败。
