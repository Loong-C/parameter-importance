# S0.2 存储布局与目录生命周期

## 目的

把 Git 内容、环境、缓存、数据、checkpoint、运行日志和结果放到明确且可控的位置，防止根盘耗尽、大文件进入 Git、不同运行互相覆盖或清理动作误伤已有资产。

## 当前依据

- 服务器项目大盘约有 2.9 TiB 可用，规定目录已存在。
- 系统根盘只余约 15 GiB，不能承载项目缓存、构建或训练产物。
- `runs`、`results`、`reports`、`operations` 和 `checkpoints` 当前基本为空，适合在新规范下启用。
- Pile shard 5 正在被既有下载进程写入，存在 `.part`、锁和元数据。
- 本机位于外接 D 盘，但大型资产的唯一权威副本仍是服务器大盘。
- 当前没有第二个已授权的大型资产副本；`DATA_ROOT` 的磁盘或文件系统故障可能同时丢失正式 checkpoint 与结果，容量充足不能替代备份。

## 前置条件

- G0-C 中的远程身份、授权路径和 `Agent/` 同步确认通过；G0-G 可在本子任务并行等待管理员处理。
- 项目大盘挂载为读写，属主和权限正确。
- 当前活动下载和锁已盘点，不把活动文件纳入清理或训练输入。

## 实施步骤

### 1. 固定存储职责

1. 将 Git 仓库限定为源代码、测试、配置、依赖声明、计划、工作日志和小型报告。
2. 将数据、模型、wheel、虚拟环境、缓存、checkpoint 和大型运行结果限定在 `DATA_ROOT`。
3. 将本机限定为轻量代码和审阅端，不建立大型资产镜像。
4. 为所有运行入口定义统一的 `DATA_ROOT` 解析方式。
5. 禁止代码在找不到 `DATA_ROOT` 时静默回退到根盘、用户主目录或系统 `/tmp`。

### 2. 明确目录用途

1. 为 `datasets` 定义只读正式输入和下载暂存的边界。
2. 为 `models` 定义固定模型、tokenizer 和初始化 checkpoint 的边界。
3. 为 `cache` 定义 Hugging Face、Torch、编译和数据集缓存的子目录。
4. 为 `envs` 和 `wheelhouse` 定义环境版本与离线包集合的关系。
5. 为 `runs` 定义单次运行的日志、解析配置和状态标记。
6. 为 `checkpoints` 定义按 run ID 隔离的完整训练状态。
7. 为 `results` 定义原始数值结果，避免与可再生报告混放。
8. 为 `reports` 定义服务器侧生成的聚合报告。
9. 为 `manifests` 定义环境、资产和结果身份清单。
10. 为 `operations` 定义运行锁、租约、心跳和维护记录。
11. 为 `tmp` 定义项目临时文件、原子写入中间文件和 bundle 中转。

### 3. 建立运行目录生命周期

1. 为每个逻辑 run 创建独立目录；新 run 创建时若目录已存在则失败。
2. 在 run 目录内按 attempt 隔离每次操作者启动，并按 session 隔离连续进程组的日志写入。
3. run 创建时写入 `CREATED`，attempt/session 启动后分别原子发布自己的 `RUNNING` 状态。
4. 正常完成时 session、attempt 和 run 依次发布成功状态，run 只允许一个最终成功状态。
5. session 失败或中止时写结构化状态；若有完整 checkpoint 且策略允许恢复，run 转为 `RESUMABLE`，否则转为最终失败/中止。
6. 恢复使用新的 attempt/session，同时保持原 run ID、checkpoint 链和 global step 连续。
7. 独立重复实验使用新的 run ID，不能借“重试”复用旧结果目录。
8. 不允许多个 rank 同时写同一个共享状态文件。

### 4. 固定缓存与临时目录

1. 把 Hugging Face、Datasets、Torch 和编译缓存显式指向 `DATA_ROOT/cache`。
2. 把项目临时文件显式指向 `DATA_ROOT/tmp`。
3. 为每次运行创建独立临时子目录。
4. 在运行结束后只清理本次清单记录的精确临时路径。
5. 若临时文件需要保留用于诊断，转移到对应 run 的诊断子目录并更新 manifest。
6. 验证未发生向根盘缓存目录的隐式写入。

### 5. 定义容量预算

1. 估算环境、模型、数据、单个 checkpoint、保存频率和最大保留数。
2. 估算每个实验的日志、参数级统计和中间结果峰值。
3. 将并行 run 数纳入总容量，而不是只估算单次运行。
4. 为大盘设置“预计峰值加安全余量”的启动条件：若预计新增量为 `E`，启动前至少需要 `E + max(0.2E, 100 GiB)` 的可用空间。
5. 设置根盘保护阈值；根盘可用空间低于 10 GiB 时禁止启动新的项目写入并上报。
6. 同时检查 inode，避免只依据字节空间。
7. 将预算结果保存为可审查的小型报告，并在配置变化时重算。

### 6. 定义保留与清理规则

1. 区分不可再生资产、正式 checkpoint、可再生缓存和临时文件。
2. 为每类产物定义最短保留期和清理责任人。
3. 正式 checkpoint 的清理由 checkpoint 索引驱动，不按通配符匹配目录名。
4. 活动 run、活动锁、`.part` 和相关进程存在时禁止清理。
5. 清理前生成精确目标清单、解析绝对路径并确认都在 `DATA_ROOT` 内。
6. 清理后记录释放空间、保留资产和可恢复性。
7. 不把 Git GC、系统日志清理或其他项目维护混入 Stage 0 gate。

### 7. 建立持久性与恢复决策

1. 将上游可按固定 revision/哈希重新获取的模型、数据、wheel 和可再生缓存标为“可替换”。
2. 将无法从 Git 或固定上游重建的正式 checkpoint、原始实验结果、人工判定和唯一运行证据标为“不可再生”。
3. 为不可再生类别记录当前副本数、共同故障域、恢复点目标、恢复时间目标和责任人。
4. 在 Stage 4 产生正式不可再生 checkpoint 前，选择并批准异盘/异主机备份目标、访问边界、加密与保留策略；未经授权不得把文件复制到 `Agent/server.md` 允许路径之外。
5. 若暂时没有获授权的第二故障域，由用户对明确列出的资产、可能损失、有效期和终止条件作出书面风险接受；不能把“服务器大盘是权威副本”表述成“不会丢失”。
6. 采用备份方案时，以 manifest 驱动增量复制，并通过抽样哈希和一次受控恢复演练验证可读性。
7. 采用风险接受时，仍保留 checkpoint 原子性、校验和、保留数与磁盘健康监测；这些措施降低逻辑损坏风险，但不冒充备份。
8. 在 Stage 4、Stage 5 和存储拓扑变化前重新审查该决策，过期的批准不能继续放行。

### 8. 建立 Git 大文件守卫

1. 定义永不允许进入 Git 的数据、权重、wheel、环境、checkpoint、缓存和大型张量类型。
2. 对其他文件设置 10 MiB 默认硬阈值；确需纳入的小型报告例外必须有明确用途、审查记录和 allowlist。
3. 同时扫描完整工作树和暂存区，不能只检查 Agent 新增文件。
4. 对压缩包和未知二进制类型即使小于阈值也要求显式审查。
5. 提供预提交/CI 可复用的检查入口，输出精确路径、大小和拒绝原因。
6. 用受控 fixture 验证禁止类型、超阈值文件和未批准例外都会使检查失败。
7. 资产 manifest、摘要和小型报告通过 allowlist 后仍需执行敏感信息扫描。

### 9. 验证权限与原子写入

1. 在每个需写目录创建小型 canary。
2. 读取并核对 canary 内容和哈希。
3. 以临时名写入后原子重命名，验证同一文件系统内发布语义。
4. 精确删除本次 canary，并确认没有遗留。
5. 验证普通运行用户不需要 `sudo` 即可完成上述操作。

## 产出

- 目录与资产生命周期规范；
- 统一路径解析和缓存配置约定；
- 运行目录状态机说明；
- Stage 4/5 checkpoint 和统计量容量预算；
- 保留、清理和磁盘保护策略；
- 可替换/不可再生资产清单、故障域分析与持久性决策记录；
- Git 大文件/禁止类型守卫及拒绝测试；
- 存储 canary 与原子写入核验报告。

## 核验标准

- 所有项目写路径都解析到 Git 仓库或 `DATA_ROOT`，没有落入根盘缓存、用户主目录或系统 `/tmp`。
- 规定的 `DATA_ROOT` 目录存在、用途唯一、权限满足实际读写需求。
- canary 的写入、读取、哈希、原子发布和精确清理全部通过。
- 活动 Pile `.part`、锁和下载进程未被修改，也未被任何数据加载器选中。
- 容量报告覆盖最大模型、并行运行数、参数级统计和 checkpoint 保留数。
- 启动检查能在空间或 inode 不足时明确失败，而不是训练中途才报错。
- 不可再生资产已有获授权的第二故障域和恢复演练证据，或存在由用户批准、范围和期限明确的单盘风险接受记录；两种路径必须二选一并可审计。
- Git 大文件守卫能拒绝数据、权重、wheel、checkpoint、未知二进制和未经批准的 10 MiB 以上文件；拒绝测试有证据。

## Gate 与后续依赖

- 路径、权限、容量、生命周期、原子写入和 Git 守卫通过形成 **G1-B 存储机制子 gate**。
- 上述持久性决策形成 **G1-D 持久性决策子 gate**；Stage 0 可通过“已验证备份”或“明确风险接受”之一满足它，但 Stage 4/5 开始前必须重新确认仍有效。
- G1-B 与 G1-D 都满足后才形成 **G1 存储 gate**。
- G1 未通过前不得下载新资产、重建环境、创建大型缓存或启动训练。
- S0.3、S0.4、S0.8、S0.9 和 S0.10 都依赖这里定义的路径与容量规则。
