# S0.12 交付、工作日志与多端同步

## 目的

把 Stage 0 的代码、配置、依赖、测试、报告和服务器资产证据收束为可审查交付物，并按 `Agent/` 规范完成本机、GitHub、服务器和运维文档的一致性核验。

## 当前依据

- `Agent/git.md` 要求每个可独立复核阶段提交并推送，用户已有修改也要一起审查处理。
- `Agent/sync.md` 要求 Git 内容三端同提交，`Agent/` 五文件两端逐文件哈希一致。
- 大型资产的权威运行副本只能留在服务器 `DATA_ROOT`，Git 中只保存小型 manifest、摘要和工作日志；G1-D 明确授权的异盘/异主机备份副本是唯一例外。
- 当前已有 `docs/mathematics.md` 用户修改，最终提交时不能静默排除。
- `DATA_ROOT` 当前是大型资产的唯一权威副本但不是第二故障域；最终交付必须如实带出 G1-D 的备份或风险接受状态。

## 前置条件

- G0–G9 均有当前有效的机器可读证据。
- 没有活动 Stage 0 测试仍在写入待封存目录。
- 所有失败、豁免和管理员处理结果已经记录。

## 实施步骤

### 1. 建立交付物清单

1. 列出依赖输入、锁文件和环境重建说明。
2. 列出配置 schema、模板、run ID、seed 和 provenance 实现。
3. 列出资产 manifest schema、审计报告和下载/校验入口。
4. 列出单卡、四卡、累积、`no_sync` 和失败 smoke test。
5. 列出日志、TensorBoard、状态机、checkpoint 和恢复实现。
6. 列出容量、性能、故障和重放报告。
7. 列出运行说明和 Stage 1 交接清单。
8. 对每项标记 Git 路径、服务器路径、大小、摘要和验收状态。

### 2. 持续更新工作日志

1. 在目标/范围冻结时写第一条 Stage 0 日志。
2. 在环境、资产、单卡、四卡、恢复和最终重放各 gate 后追加记录。
3. 每条记录包含目标、实际修改、验证、退出状态、证据路径和风险。
4. 失败和中止保留原记录，后续以新条目更正，不重写历史。
5. 使用北京时间并记录当前分支和提交。
6. 不把秘密、临时 CDN 地址或大型产物内容写入日志。

### 3. 审查完整工作树

1. 查看所有已跟踪修改、删除和未跟踪文件。
2. 审查用户已有 `docs/mathematics.md` 修改并运行适合该文档的检查。
3. 确认 Stage 0 生成的大型文件、环境、缓存和 checkpoint 没有进入仓库。
4. 运行格式、空白、链接、敏感信息和大文件检查。
5. 检查测试报告中没有本地绝对秘密路径或临时下载地址。
6. 对任何冲突、不可验证用户修改或大型文件保留原内容并上报，不静默省略。

### 4. 划分可复核提交

1. 基线/规范和环境声明形成独立提交。
2. 配置、日志和 checkpoint 基础设施形成独立提交。
3. 单卡/四卡测试和报告形成独立提交。
4. 资产 manifest、运行说明和最终 gate 形成收尾提交。
5. 每次提交前更新工作日志并检查暂存差异。
6. 提交信息准确说明成果和 gate，不使用含糊描述。
7. 若采用阶段分支，在合并前完成全部测试和审阅。

### 5. 推送 GitHub

1. 推送前只读核对远端目标分支，确认没有新提交。
2. 遇到非快进时停止，检查提交图并作明确 merge/rebase 决策。
3. 不使用强制推送。
4. 推送后再次读取 GitHub 分支 HEAD。
5. 大改动或实验口径变化通过 PR 审阅；简单快进是否直接合并需在日志中说明。

### 6. 同步服务器 Git 仓库

1. 确认服务器工作树干净、分支和 HEAD 符合预期。
2. 从本机创建并验证只包含目标提交关系的 Git bundle。
3. 将 bundle 放入 `DATA_ROOT/tmp` 的精确路径。
4. 服务器获取 bundle 后只做快进。
5. 不能快进、工作树有改动或提交分叉时立即停止。
6. 核对服务器 HEAD 与本机/GitHub 完全一致。
7. 只在三端一致后精确删除本次本机和服务器 bundle。

### 7. 同步 `Agent/` 文档

1. 再次列出本机和服务器五文件集合。
2. 只同步已审查的规定文件。
3. 逐文件核对大小和 SHA-256。
4. 不把 `Agent/` 加入 Git，也不删除未经确认的服务器文件。
5. 将最终哈希表写入工作日志。

### 8. 封存服务器专属资产证据

1. 对环境、wheelhouse、模型、数据、checkpoint 和报告生成最终 manifest 摘要。
2. 每项记录绝对路径、时间、关联提交、大小、哈希/revision 和状态。
3. 验证服务器大盘是大型资产的唯一权威运行副本，同时记录这不等同于备份。
4. 对不可再生产物关联 G1-D 决策：记录获授权第二故障域、最近恢复演练，或用户批准的单盘风险范围与有效期。
5. 本机若有临时小型结果副本，立即核对并纳入 Git 或按已授权的精确清单处理；不得借收尾步骤扩大复制或删除范围。
6. 汇总 gate 状态：全部为 `PASS` 时生成新的 Stage 0 `READY`；存在合规条件接受时只能生成 `READY_WITH_APPROVED_EXCEPTIONS`，并内嵌例外范围、批准者和失效日期。
7. readiness 标记只在所有 gate 按统一状态规则满足后原子发布，名称和内容不得覆盖旧历史 `READY`。

### 9. 编写 Stage 1 交接

1. 列出 Stage 1 可直接复用的环境 ID、配置模板和资产 ID。
2. 列出单卡/四卡数值容差和参考报告。
3. 列出固定调试 fixture、Pythia 14M 和数据范围。
4. 列出 loss reduction、梯度累积和 `no_sync` 已验证语义。
5. 列出尚未解决但不阻断 Stage 1 的风险。
6. 明确 Stage 1 仍需证明参数重要性实现正确，不能引用 Stage 0 代替。

## 产出

- Stage 0 最终交付物与证据清单；
- 持续更新的中文工作日志；
- 可审查的阶段提交与必要的 PR；
- 本机/GitHub/服务器三端一致性记录；
- 本机/服务器 `Agent` 五文件哈希表；
- 服务器大型资产最终 manifest 摘要；
- 不可再生产物的 G1-D 持久性/风险接受状态摘要；
- 新的 Stage 0 `READY` 或 `READY_WITH_APPROVED_EXCEPTIONS` 标记和 Stage 1 交接文档。

## 核验标准

- G0–G9 均有当前有效报告，任何硬 gate 不得以豁免冒充通过；性能/容量例外必须显示为 `APPROVED_EXCEPTION/CONDITIONALLY_ACCEPTED`。
- 完整工作树已审查；用户已有修改得到保留、验证、记录和一致处理。
- Git 中没有数据、模型、wheel、环境、checkpoint、缓存或大型原始结果。
- 所有硬正确性与安全测试均为 `PASS`，不存在未批准的 `FAIL` 或硬 gate skip；获批性能/容量阈值保持 `APPROVED_EXCEPTION`，并与全部失败、跳过和退出状态一起进入总报告。
- 本机 HEAD、GitHub 分支 HEAD 和服务器 HEAD 完全相同。
- 服务器工作树干净，本次临时 bundle 已按精确路径清理。
- 两端 `Agent/` 文件集合完全相同且五个 SHA-256 逐一一致。
- 每项服务器专属资产都有路径、关联提交、大小、哈希/revision 和验收状态。
- G1-D 状态能定位到获授权备份与恢复证据，或定位到仍有效的用户风险接受；不得把单盘权威副本描述为已备份。
- 新 readiness 标记由最终报告原子生成，且能反向定位全部证据；带例外状态不得发布普通 `READY`。
- 独立执行者能依据交接文档开始 Stage 1，不需要猜测环境或资产身份。

## Gate 与后续依赖

- 本子任务通过形成 **G10 最终同步 gate**。
- 只有 G0–G10 的硬项全部 `PASS`，且各 gate 为 `PASS` 或合规的 `CONDITIONALLY_ACCEPTED`，才能宣布 Stage 0 完成并进入 Stage 1。
- 任一三端提交、`Agent` 哈希、资产 manifest 或 readiness 证据不一致时，状态只能是“收尾中”或“阻塞”。
