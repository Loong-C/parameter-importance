# S0.10 容量评估、运行保护与故障处理

## 目的

用分析预算和实测数据证明四卡训练、日志、按参数形状构造的容量占位张量与 checkpoint 能在当前服务器资源内稳定运行，并建立不抢占他人设备、不耗尽磁盘、不留下孤儿进程的运行保护机制。这里的占位张量只验证资源上界，不实现或验证 Stage 1–3 的重要性算法。

## 当前依据

- 服务器有约 1 TiB 内存和约 2.9 TiB 项目大盘余量，但根盘只余约 15 GiB。
- 当前运行时只枚举 7/8 张 GPU，且一张可见卡有不可纠正 ECC；这是运维硬阻塞。
- 当前四卡候选拓扑没有活动 NVLink 证据，通信性能需要实测。
- 文件描述符软上限为 1024，数据加载 worker 数不能凭经验无限增加。
- Pile 全量下载仍在运行，网络、磁盘和 SSH 波动是现实约束。
- Stage 5 将使用 410M 并保存高频 checkpoint 和参数级统计，存储压力可能大于模型训练本身。

## 前置条件

- G0-G 与 G5–G7 已通过并提供可测量的单卡、四卡、日志和 checkpoint 实现。
- GPU 管理员问题已解决，候选四卡健康。
- G1 已定义容量安全余量、根盘保护和清理规则。
- `README.md` 4.1 节的统一测量协议版本已经冻结。
- 核心保护和分析预算可以先做；160M 实测必须等待 G3-S4，410M 实测必须等待 G3-S5。

## 实施步骤

### 1. 建立容量估算器

1. 根据参数量和 dtype 估算模型权重字节数。
2. 分别估算梯度、master weights、optimizer state 和 scheduler state。
3. 根据 batch、sequence length、层数和 checkpointing 策略估算 activation。
4. 根据每个参数的真实 shape/dtype 单独估算 raw、signed、positive、negative 等参数级累计张量。
5. 估算每个 checkpoint 的共享和 per-rank 状态。
6. 估算日志、TensorBoard、参数级结果和报告的总量。
7. 将保存频率、保留数、seed 数和并行 run 数纳入总预算。
8. 对 14M、31M、160M 和 410M 读取各自固定 config/参数 shape 分别输出预算，不从某个规模线性外推；160M/410M 还必须在资产 gate 后实测。
9. 为估算公式建立小张量解析 fixture，并让实测报告回填估算误差。
10. 冻结版本化“参数状态容量包络”，逐项列出 full-parameter resident buffer、瞬时归约/序列化 buffer、元素数量、dtype、生命周期、是否同时存活和保存频率。
11. 容量包络至少覆盖 raw、signed、positive、negative、U-statistic 所需矩状态、路径积分累计、分布式归约暂存和 checkpoint 序列化暂存；这里只按声明的 shape/dtype 执行 synthetic 更新，不断言任何公式正确。
12. 同时冻结版本化“计算/通信工作包络”，逐项列出每个 optimizer step 允许的前向/反向与梯度求值次数、重算次数、collective 类型/次数/字节量、CPU 聚合和序列化工作量。
13. 用不包含重要性数学的 synthetic workload driver 复现该工作包络的额外计算与通信上界，并把其 wall-clock、scaling 和 checkpoint 暂停纳入预算。

### 2. 实测显存与主存

1. 按统一协议在单卡上执行 5 个 warmup step、20 个测量 step 和三个全新进程重复，分阶段记录加载、前向、反向、optimizer step 和 checkpoint 的峰值。
2. 用相同协议在四卡上记录每 rank 峰值、不平衡程度和三次重复中的最高峰值。
3. 分别测试 FP32 语义配置和 BF16 正式配置。
4. 测量不同 microbatch、accumulation 和 sequence length。
5. 使用容量包络中声明的真实参数 shape/dtype、同时存活关系和生命周期，分配、清零和做 synthetic 更新以测量增量；不在这里实现 raw/U-statistic/路径积分等重要性公式。
6. 测量 checkpoint 保存时 CPU 内存峰值，防止 rank 0 聚合造成瞬时翻倍。
7. 记录 OOM 时最后成功配置，禁止自动缩 batch 后继续却不改变配置身份。
8. G3-S4 通过后，用 160M 的 Stage 4 候选训练配置、正式日志、实际 checkpoint cadence、参数状态容量包络和计算/通信工作包络实测完整窗口。
9. G3-S5 通过后，用 410M 的 Stage 5 候选四卡训练配置、正式日志、高频 checkpoint cadence、参数状态容量包络和计算/通信工作包络实测完整窗口；二者不得以小模型外推代替。

### 3. 实测吞吐与通信

1. 按统一协议的 10 个 warmup、30 个测量 step 和三个全新进程重复记录单卡 tokens/s、samples/s 和 step time。
2. gate 默认使用 strong scaling：固定每个 optimizer step 的有效 global token 数、sequence length、模型、精度和 optimizer 语义，通过 per-device batch/accumulation 调整单卡与四卡配置。
3. 将吞吐定义为单位 wall-clock 的有效非 padding、非 ignore token 数；同时报告 samples/s，但不把 padding token 计入 scaling gate。
4. 用相同 fixture 和协议记录四卡总吞吐，并按 `四卡有效 tokens/s ÷ (4 × 单卡有效 tokens/s)` 计算 strong-scaling efficiency。
5. 若另做 weak scaling，必须固定每卡有效工作量、单独命名和报告，不与 strong-scaling gate 混用。
6. 记录 NCCL collective 基线及其波动。
7. 区分数据读取、计算、通信、日志和 checkpoint 时间。
8. 在当前拓扑下比较候选四卡组，但不占用其他用户设备或跨越未批准设备。
9. 将 validated baseline 与硬件/内核/环境 ID 绑定，环境漂移后自动失效。

### 4. 验证数据加载与文件描述符

1. 从少量 worker 开始测量吞吐、内存和打开文件数。
2. 逐级增加 worker，找到吞吐不再明显提升的点。
3. 确认峰值打开文件数低于软上限并保留余量。
4. 验证多 rank worker 总数，而不是只看单 rank。
5. 验证 reader 不为每个样本重复打开大文件。
6. 若必须提高单进程限制，先证明必要性并使用非系统级、可记录的启动配置；不得擅改全局限制。

### 5. 建立 GPU 使用保护

1. GPU 列表由操作者或调度配置显式提供。
2. 启动前检查目标 GPU 的进程、显存、ECC、温度和拓扑。
3. 将 GPU UUID/PCI 映射写入项目级租约和 provenance。
4. 租约只保护本项目并发，不能冒充集群调度器或强占其他用户资源。
5. 对正式四卡运行取得资源所有者/管理员的可核验确认或预约，并在创建进程组前再次检查，降低检查与启动之间的竞态窗口。
6. 发现外部进程或未知占用时退出，不终止他人进程。
7. 运行中监测设备消失、ECC 增长和 Xid/通信错误信号。
8. 结束后释放项目租约并确认 GPU 上下文归零。

### 6. 建立运行前检查

1. 检查 G0–G7 对应证据未过期。
2. 检查代码、配置、环境和资产摘要匹配。
3. 检查 GPU 健康、可用性和四卡映射。
4. 检查数据游标覆盖和活动下载边界。
5. 检查大盘/根盘空间、inode 和 checkpoint 预算。
6. 检查 G1-D 持久性决策仍有效，并在报告中区分“有第二故障域”和“已接受单盘风险”。
7. 检查文件描述符、内存和输出目录碰撞。
8. 检查没有冲突的项目锁或 stale 状态未处理。
9. 所有检查通过后才发布 running 状态。

### 7. 建立故障分类与响应

1. OOM：保留配置与峰值证据，终止本次 attempt，不静默改变 batch。
2. NCCL/超时：让所有 rank 有界退出，保存 rank 日志和设备映射。
3. ECC/设备消失：立即停止使用目标设备，上报管理员，废止相关硬件 gate。
4. 磁盘不足：停止新 checkpoint，优先安全结束或从上一完整 checkpoint 恢复，不临时删除未知文件。
5. JSONL/状态真值写入失败：生成明确错误并停止；TensorBoard 派生层失败只告警并保留 JSONL 真值。
6. checkpoint 损坏：回退到上一个完整对象并保留损坏证据。
7. SSH 断开：服务器训练进程的生命周期与交互会话解耦，后续通过状态和心跳判断，不重复启动。
8. 下载故障：沿用既有锁、断点续传和监督链路，不与训练脚本混合修复。
9. `DATA_ROOT` 设备或文件系统故障：停止宣称本盘 checkpoint 可恢复；若有获授权的第二故障域则按恢复 runbook 取回并校验，否则按 G1-D 风险接受记录明确报告不可恢复损失，不把同盘多副本冒充备份。

### 8. 验证受控故障

1. 在 smoke 配置中触发可控 OOM 或预检内存超限。
2. 触发单 rank 异常并验证全组退出。
3. 模拟日志/磁盘 canary 写失败。
4. 模拟 stale 心跳但仍有进程，验证系统不会误删锁。
5. 模拟 SSH 客户端中断，验证不会重复启动同一 run。
6. 每个场景记录检测、响应、证据和安全重试条件。

## 产出

- 14M/31M/160M/410M 的显存、内存、存储和 checkpoint 预算；
- 版本化参数状态容量包络、计算/通信工作包络、synthetic fixture 与估算误差报告，不含重要性算法实现；
- 单卡/四卡吞吐、scaling 和 NCCL 性能基线；
- 160M 的 G8-S4 与 410M 的 G8-S5 独立实测报告；
- 数据加载 worker/文件描述符基线；
- GPU 租约、运行前检查、心跳和运行后检查机制；
- 故障分类、停止条件和恢复 runbook；
- 受控故障演练报告。

## 核验标准

- 按统一协议三次重复观测到的最高每卡峰值显存不超过可用显存的 80%；若需要更高占用，只能按统一协议记录单独稳定性证据与用户批准的 `APPROVED_EXCEPTION`，不能写成达到原阈值。
- 四卡各 rank 显存差异可解释，无单 rank 异常聚合导致的 OOM 风险。
- 项目大盘启动前若预计新增量为 `E`，可用空间至少为 `E + max(0.2E, 100 GiB)`；根盘不少于 10 GiB。
- 峰值打开文件数不超过软上限的 70%，或已有经过记录的非系统级提升方案。
- 正式日志相对最小日志的三组成对重复吞吐开销不超过 10%，checkpoint 保存暂停时间进入训练预算；例外遵守统一协议的批准和标记规则。
- 四卡性能基线与环境/拓扑绑定；短 smoke 低于基线 70% 时告警并触发完整统一协议复验，只有完整复验仍低于 70% 时阻断。
- synthetic accumulator 只证明 shape/dtype 资源上界，不得作为任何重要性方法的正确性证据。
- 160M 与 410M 的正式候选配置分别覆盖训练、正式日志、实际 checkpoint cadence、resident/transient buffer 以及额外前反向/重算/collective/序列化工作包络并完成真实资产实测，不能用分析估算或较小模型外推冒充。
- GPU 健康异常、外部占用、磁盘不足、未知资产和输出碰撞均在启动前失败。
- 所有受控逻辑故障有界退出、无孤儿进程、无错误成功标记，且上一完整 checkpoint 可用；整盘/文件系统故障只按获授权第二故障域恢复或明确报告已接受的数据丢失风险。

## Gate 与后续依赖

- 核心运行保护、统一协议和 synthetic shape 容量测试通过形成 **G8-C**。
- G3-S4 通过后，160M 的 Stage 4 候选配置连同正式日志、实际保存频率、完整参数状态容量包络和计算/通信工作包络完成真实实测才形成 **G8-S4**。
- G3-S5 通过后，410M 的 Stage 5 候选四卡配置连同正式日志、高频 checkpoint、完整参数状态容量包络和计算/通信工作包络完成真实实测才形成 **G8-S5**。
- **G8 容量与运维总 gate** 在 G8-C、G8-S4、G8-S5 全为 `PASS` 时记为 `PASS`；若仅有按统一规则批准的性能/容量 `APPROVED_EXCEPTION` 且没有失败、阻塞或过期项，则记为 `CONDITIONALLY_ACCEPTED`。
- Stage 1–3 的最终重要性实现若改变计算图、梯度求值/前反向/重算次数、collective schedule、序列化工作量，或增加 buffer 数量、dtype、同时存活集合、保存内容或生命周期并超出任一已测包络，G8-S4/G8-S5 自动失效，必须按新包络重验。
- 任何硬件、内核、依赖锁、拓扑或正式配置变化都会使相关性能/容量证据失效并要求重验。
