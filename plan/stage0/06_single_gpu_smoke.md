# S0.6 单卡训练 smoke test

## 目的

在不引入多卡通信复杂度的情况下，先验证设备选择、资产加载、数据读取、前向、反向、优化器、混合精度、日志和最小 checkpoint 能形成一个完整小闭环。

## 当前依据

- 服务器现有环境曾完成 160M BF16 forward/backward，但旧证据对应旧内核和 8 卡枚举。
- 当前 GPU 枚举和 ECC 存在异常，必须先由 G0-G GPU 健康子 gate 解除。
- Pythia 14M step0 与最小 Pile 前缀已有历史资产，可作为资产重验后的真实模型 smoke test。
- 当前 `main` 没有现行训练代码，不能把旧归档运行结果当作新入口的验证。

## 前置条件

- G2 环境 gate、G3-S1 资产 gate 和 G4 配置 gate 通过。
- G0-G 已通过，目标单卡 ECC/健康和占用检查通过。
- 所有缓存、临时文件和输出路径已指向 `DATA_ROOT`。
- `README.md` 4.1 节的统一测量协议版本已经冻结并进入 resolved config/provenance。

## 实施步骤

### 1. 建立最小训练组件

1. 实现运行前环境与资产检查组件。
2. 实现显式设备选择组件，禁止默认使用“第一张可见卡”而不记录映射。
3. 实现模型/config/tokenizer 加载组件。
4. 实现固定样本 ID 的最小数据读取组件。
5. 实现 loss 计算和 reduction 组件。
6. 实现反向传播、梯度清零和 optimizer step 组件。
7. 实现有限值、梯度范数和参数更新检查组件。
8. 实现结构化指标和状态写入接口。
9. 实现最小 checkpoint 保存接口，完整恢复在 S0.9 扩展。

### 2. 建立两级 smoke 配置

1. 设计无外部大型资产的极小确定性模型/数据配置，用于快速代码回归。
2. 设计 Pythia 14M + 固定真实 token batch 配置，用于资产与模型栈验证。
3. 两级配置使用相同训练组件和日志 schema。
4. 两级配置都限制步数、输出量和显存占用，不能演变成正式实验。

### 3. 验证 FP32 确定性路径

1. 关闭 dropout 等额外随机性。
2. 固定模型初始化、样本 ID、batch 顺序和 optimizer。
3. 运行一次前向并保存 loss 摘要。
4. 运行一次反向并保存选定张量的梯度摘要。
5. 执行一次 optimizer step 并保存参数更新摘要。
6. 从相同初始状态重复运行。
7. 比较两次 loss、梯度和参数更新。

### 4. 验证短程训练闭环

1. 使用统一测量协议规定的固定极小过拟合 fixture、50 个 optimizer step 和三次全新进程重复验证 loss 下降。
2. 每步记录 global step、样本 ID、loss、梯度范数、学习率和显存。
3. 验证 step 单调、样本顺序符合配置、参数确实变化。
4. 验证没有 NaN、Inf、空梯度或意外冻结参数。
5. 验证运行结束写入唯一成功状态。

### 5. 验证 BF16 路径

1. 使用 A100 支持的 BF16 autocast 配置。
2. 运行 Pythia 14M 前向、反向和 optimizer step。
3. 检查 loss、梯度和参数更新为有限值。
4. 记录峰值显存和 step 时间。
5. 不用 BF16 与 FP32 的逐元素严格一致作为 gate；只把差异作为诊断指标。

### 6. 验证数据边界

1. 在日志中记录每个 batch 的稳定样本 ID 或样本摘要。
2. 验证 sequence length、attention mask、labels 和 ignore index。
3. 验证 tokenizer 与模型 vocab 一致。
4. 验证训练游标没有超过资产 manifest 覆盖范围。
5. 故意提供越界游标，确认启动前失败。

### 7. 验证资源稳定性

1. 按统一测量协议记录启动前、模型加载后、warmup 后和 20 个测量 step 的显存，并做三次全新进程重复。
2. 区分 allocated 与 reserved memory。
3. 把测量窗口拆成四个等长子窗口，检查 allocated memory 没有跨四个窗口持续单调增长。
4. 记录 CPU 内存、文件描述符和数据加载 worker 状态。
5. 结束后确认没有残留训练进程或占用的 GPU 上下文。

### 8. 验证失败路径

1. 提供非法资产 ID，确认在加载前失败。
2. 提供非有限 loss 测试条件，确认立即中止并写失败状态。
3. 模拟输出目录已存在，确认不覆盖。
4. 模拟最小 checkpoint 写入失败，确认成功状态不会发布。
5. 确认失败日志保留根因、最后有效 step 和安全重试说明。

## 产出

- 单卡 smoke test 入口和两级配置；
- 最小训练组件及对应单元测试；
- FP32 确定性报告；
- Pythia 14M BF16 单卡报告；
- 数据边界、资源稳定性和失败路径报告；
- 单卡运行的 resolved config、provenance、日志和最小 checkpoint。

## 核验标准

- 所有运行只使用显式选择且通过健康检查的单张 GPU。
- FP32 两次确定性运行的 loss、梯度和更新张量满足 `atol=1e-6, rtol=1e-5`；模型初始化摘要和样本 ID 完全相同。
- 三次极小过拟合运行的最后 10 步 loss 中位数都至少比最初 10 步低 5%，所有 step 的 loss、梯度范数和更新均有限。
- Pythia 14M BF16 前向、反向和 optimizer step 完成，无 CUDA/ECC/Xid/OOM 错误。
- 样本游标、sequence length、vocab、labels 和 manifest 范围全部一致；越界输入被拒绝。
- 三次显存测量中，allocated memory 均未跨四个等长稳定子窗口持续单调增长，最后窗口中位数不超过首个稳定窗口中位数的 105%；不满足时必须定位并重验，风险说明不能替代 G5。
- 正常运行只有一个成功状态；所有故意失败路径只有结构化失败状态且无残留进程。
- 本机 CPU 测试只验证配置和极小逻辑，不被计作 G5 的 GPU 证据。

## Gate 与后续依赖

- 本子任务通过形成 **G5 单卡 gate**。
- G5 未通过不得进入四卡 DDP 语义验证。
- Stage 1 的参数重要性代码必须复用本子任务的 loss、设备、数据和训练 step 组件，避免另建不一致路径。
