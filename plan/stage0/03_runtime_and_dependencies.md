# S0.3 运行时、依赖锁定与环境重建

## 目的

把当前可用的服务器虚拟环境转化为可解释、可重建、可离线验收的正式环境，并为没有 NVIDIA GPU 的本机定义轻量开发环境，避免跨平台解析再次遗漏 Linux/CUDA 依赖。

## 当前依据

- 服务器现有 venv 使用 Python 3.12.3，89 个锁定分发包与现场版本一致，`pip check` 通过。
- 现有 wheelhouse 约 3.5 GiB，并有 SHA-256 清单。
- PyTorch wheel runtime 为 CUDA 12.6；系统驱动支持更高版本，系统 toolkit/`nvcc` 为 12.9 且不在默认 PATH。
- 本机只有 Python 3.12.4 和 CPU PyTorch 2.10.0，缺少完整项目栈。
- DeepSpeed 与 W&B 不在现有环境；Accelerate 和 TensorBoard 已在服务器环境中。
- 旧环境快照的内核/GPU 枚举已漂移，需要重新生成运行环境 manifest。

## 前置条件

- G1 通过，`envs`、`wheelhouse`、`cache` 和 `tmp` 的路径已固定。
- GPU 枚举/ECC 异常已上报。环境文件和 CPU 导入可在 G0-C/G1 后先处理，但 GPU smoke 必须等待 G0-G。
- 不使用 `sudo`，不修改系统 Python、系统驱动或系统 CUDA。

## 实施步骤

### 1. 冻结运行时方案

1. 明确服务器 Python 主版本和可接受的补丁版本范围。
2. 分别记录 NVIDIA 驱动、系统 CUDA toolkit、PyTorch wheel CUDA runtime、cuDNN 和 NCCL。
3. 选择原生 PyTorch DDP/`torchrun` 作为分布式真值层。
4. 保留 Accelerate 作为已锁定工具，但不让其隐藏 Stage 0 的梯度与同步语义测试。
5. 明确 Stage 0 不引入 DeepSpeed；只有后续容量证据表明必要时才单独评审。
6. 选择本地结构化日志和 TensorBoard，明确 W&B 不是必需依赖。
7. 明确不编译自定义 CUDA 扩展；若以后需要，另行冻结编译器、ABI 和 toolkit 组合。

### 2. 拆分依赖声明职责

1. 建立平台无关的基础依赖集合。
2. 建立服务器 Linux/CUDA 专属依赖集合。
3. 建立本机 CPU/开发/文档/测试依赖集合。
4. 让锁文件记录解析平台、Python 版本、索引来源和生成工具版本。
5. 避免用 Windows 主机的 marker 语义直接推导 Linux CUDA 完整依赖。
6. 为每份输入声明和锁文件生成 SHA-256。
7. 明确何种变更必须重新锁定，禁止手工只改某个传递依赖而不重解。

### 3. 审计现有服务器环境

1. 对锁文件的每个分发包核对名称和精确版本。
2. 检查是否存在锁外包、可编辑安装或本地未记录 wheel。
3. 运行依赖一致性检查。
4. 记录现场 freeze、Python 路径和 site-packages 位置。
5. 验证核心模块导入和版本字段。
6. 验证 PyTorch 能识别 CUDA runtime，但在硬件 gate 前不把设备数作为通过证据。
7. 将结果写入新的环境审计 manifest，不覆盖历史文件。

### 4. 审计 wheelhouse

1. 对 wheelhouse 文件集合与清单逐项比较。
2. 核对每个 wheel 的大小和 SHA-256。
3. 确认锁文件中的每个包都有适配 Linux、Python 和架构的离线候选。
4. 确认 wheelhouse 没有无法解释的额外文件。
5. 对索引来源和许可证/分发来源做最小记录。
6. 将 wheelhouse 清单与锁文件摘要关联。

### 5. 做非破坏性离线重建

1. 在 `DATA_ROOT/envs` 下选择新的、带版本标识的候选路径。
2. 保持现有已验收 venv 不变，禁止原地卸载或升级。
3. 使用唯一、空的候选环境和隔离的安装缓存，避免复用主机上未纳入清单的 wheel。
4. 将安装器限制为只解析已验真的 wheelhouse，禁止使用在线索引、依赖在线回退或未记录的本地缓存。
5. 在受控网络阻断或连接审计条件下完成安装，并保存零外连尝试的证据；仅清空代理变量不能作为离线证明。
6. 运行依赖一致性检查和核心导入 smoke test。
7. 生成候选环境现场 freeze，并与锁文件和现有环境比较。
8. 从 wheelhouse 的隔离副本中移除一个必需 wheel，验证负向重建明确失败且不会转而联网补包。
9. 通过后原子更新项目“推荐环境”引用；未通过则保留诊断，继续使用旧环境。
10. 记录重建耗时、占用空间、网络隔离方式和失败恢复路径。

### 6. 建立本机轻量开发环境

1. 鉴于本机没有 Conda、Mamba 或 uv，使用 Python 标准虚拟环境作为轻量隔离层，并确保环境目录不进入 Git。
2. 不安装 CUDA、NCCL 或 Linux 专属 wheel。
3. 安装或锁定配置解析、schema、测试、静态检查和 CPU 小张量测试所需的最小集合。
4. 将 GPU、NCCL、BF16 性能测试标记为服务器专属。
5. 让服务器专属测试在本机明确 skip，而不是假通过。
6. 验证本机环境不会下载或保存大型模型/数据。

### 7. 生成环境身份

1. 为依赖输入、锁文件、wheelhouse、现场 freeze 和环境路径分别记录摘要。
2. 记录 OS、内核、glibc、Python、驱动、toolkit、PyTorch runtime、cuDNN 和 NCCL。
3. 记录 GPU 枚举健康结果的引用，而不是把硬件状态混入不可变依赖锁。
4. 生成单一 `environment_id`，由稳定字段计算。
5. 每次运行把 `environment_id` 和现场漂移检查结果写入 provenance。

## 产出

- 平台无关、服务器 CUDA 和本机开发三层依赖声明；
- 精确锁文件及其生成说明；
- wheelhouse 完整性报告；
- 非破坏性离线重建、零外连审计和缺 wheel 负向报告；
- 本机轻量开发环境说明；
- 新的服务器环境 manifest 和稳定 `environment_id`；
- 分布式与追踪工具的方案决策记录。

## 核验标准

- 锁文件中的包全部有精确版本，解析平台和 Python 版本明确。
- 服务器现场环境与服务器锁完全匹配；任何额外包都已解释并纳入或移除于新候选环境。
- `pip check` 或等价依赖检查无错误。
- wheelhouse 文件集合、大小和 SHA-256 全部通过，能覆盖离线重建的每个包。
- 候选环境在唯一空路径、隔离缓存和禁止联网条件下只依靠 wheelhouse 重建成功，连接审计为零外连尝试。
- 缺失任一必需 wheel 的负向演练会明确失败，且不会访问在线索引或其他主机缓存。
- 核心 CPU 导入与版本核对通过；GPU 导入在硬件 gate 后逐卡重验。
- 重建过程没有写入系统根盘的项目缓存或环境目录。
- 同一锁、wheelhouse 和平台生成相同现场 freeze；差异必须有报告。
- 本机测试不会把 CPU PyTorch 结果冒充服务器 CUDA 结果。

## Gate 与后续依赖

- 本子任务通过形成 **G2 环境 gate**。
- G2 是 S0.6 的直接前置条件；S0.7 通过其 G5 前置传递依赖 G2。
- 若只完成依赖审计、尚未完成离线重建，则只能标记“环境候选可用”，不能通过 G2。
- 自定义 CUDA 扩展一旦进入范围，必须新增编译工具链 gate，不能沿用当前 G2 自动放行。
