# S0.1 基线、安全边界与多端状态

## 目的

建立一个可信、可重复采集的项目起点，并先解决会污染后续结论的访问、同步、硬件和权限问题。这个子任务只做只读盘点、必要的文档同步修复和管理员协调，不安装依赖、不启动训练、不修改系统配置。

## 当前依据

- 本机、GitHub 和服务器 `main` 当前提交一致，但本机有用户未提交修改。
- `Agent/worklogs.md` 两端不一致，其他四份 `Agent` 文档一致。
- 服务器底层驱动目录可见 8 张 A100，NVML/PyTorch 只见 7 张；一张可见 GPU 有不可纠正 ECC 记录。
- 旧 `READY` 使用不同的内核和 8 卡枚举结果，不能代表当前状态。
- 服务器根盘余量较小，项目大盘空间充足。

## 前置条件

- 只使用 `Agent/remote_access.md` 已定义的 SSH 别名和既有公钥链路。
- 不读取或记录密钥、密码、令牌、Cookie、完整签名 URL、临时 CDN 地址或 SSH 端口细节。
- 不主动断开隧道测试恢复，不修改 `start.cmd`、SSH 配置或网络拓扑。
- 只在服务器仓库和 `DATA_ROOT` 两个授权根目录内进行后续写操作。

## 实施步骤

### 1. 固定任务范围

1. 将 Stage 0 的包含项、排除项和完成 gate 写入本次工作日志。
2. 标明本轮是“计划执行/基础设施建设”，不是正式训练实验。
3. 标明本机、lab-pc、服务器和 GitHub 各自职责。
4. 标明需要管理员处理的事项与 Agent 可自行完成的事项。
5. 确认不通过修改系统驱动、系统 CUDA、DNS 或 SSH 来规避问题。

### 2. 验证远程访问链路

1. 从本机对 `lab-pc` 做非交互连接验证。
2. 记录 lab-pc 的主机名、用户身份、初始目录和退出状态。
3. 从本机对 `sophgo13-via-lab` 做非交互连接验证。
4. 记录服务器主机名、用户身份、初始目录和退出状态。
5. 确认两次验证都没有回退到密码认证。
6. 若连接失败，只按既有隧道恢复说明处理；持续失败则停止本子任务。

### 3. 采集仓库起点

1. 记录本机当前分支、HEAD、工作树状态和远端地址类别。
2. 只读查询 GitHub 目标分支 HEAD，避免把本机缓存引用当作远端事实。
3. 记录服务器当前分支、HEAD 和工作树状态。
4. 对比三端提交关系，确认不存在分叉或未知服务器修改。
5. 完整审查本机用户已有修改，记录其来源、内容和验证需求。
6. 决定 Stage 0 使用当前分支还是新建阶段分支；若创建分支，先记录基线提交。
7. 不隐藏、还原或拆开用户已有修改来制造“干净”提交。

### 4. 修复 `Agent/` 同步债务

1. 列出本机和服务器 `Agent/` 文件集合，确认都只有规定的五份文档。
2. 对每份文件计算 SHA-256 和字节大小。
3. 对不一致文件做逐行语义比较，区分内容变更与换行差异。
4. 根据变更来源确定权威版本；不在未审查时直接覆盖任一端。
5. 只复制经过确认的五个精确文件，不使用通配清理服务器目录。
6. 复制后重新核对文件集合、大小和哈希。
7. 把差异原因、选择依据和最终哈希写入工作日志。

### 5. 采集服务器系统基线

1. 记录发行版、内核、glibc、CPU 拓扑、内存和 swap。
2. 记录根盘与项目大盘的文件系统、空间和 inode。
3. 记录 `DATA_ROOT` 属主、权限和读写状态。
4. 记录进程文件描述符、锁定内存等与数据加载/NCCL 有关的限制。
5. 记录 NVIDIA 驱动版本、系统 CUDA toolkit、PyTorch CUDA runtime，保持三者字段独立。
6. 生成机器可读的基线快照，并在工作日志中只引用路径与摘要。

### 6. 处理 GPU 枚举与健康异常

1. 分别记录 PCI/驱动层、NVML 和 PyTorch 看到的 GPU 数量与总线映射。
2. 对每张被枚举设备记录型号、显存、volatile/aggregate ECC、retired pages、row-remap pending/failure、温度、占用和 MIG 状态；驱动支持时同时记录 Xid/恢复动作摘要。
3. 将“底层 8 张、运行时 7 张”和不可纠正 ECC 作为硬件阻塞上报管理员。
4. 不由项目脚本执行 GPU reset、驱动重载、节点重启或 ECC 清除。
5. 管理员处理后重新采集同一组字段。
6. 验收路径 A 是修复为 PCI/驱动、NVML、PyTorch 8/8/8 一致枚举。
7. 验收路径 B 是管理员批准稳定隔离/降配，提供期望可用 GPU 的 PCI/UUID 白名单和被排除设备处置结论；NVML、PyTorch 必须对该白名单映射一致。
8. 未解释的 8/7 差异不属于路径 B，仍然阻断。
9. 在期望设备集合一致且候选四卡健康之前，不运行 CUDA、NCCL 或 DDP gate。

### 7. 封存基线证据

1. 将机器可读系统快照放入服务器 `manifests` 或本次受控运行目录。
2. 将小型摘要、schema 和核验报告纳入 Git。
3. 为快照记录生成时间、采集器版本、Git 提交和 SHA-256。
4. 明确旧 `READY` 为历史证据，不覆盖、不删除，也不复用为当前结果。

## 产出

- Stage 0 基线工作日志条目；
- 本机、GitHub、服务器仓库状态摘要；
- 本机/服务器 `Agent` 文件集合与哈希报告；
- 服务器系统、存储、限制、驱动和 GPU 健康快照；
- GPU 异常管理员处理单或明确的新硬件基线确认；
- 当前可继续项、阻塞项和责任人的清单。

## 核验标准

- `lab-pc` 与服务器均能通过既有别名非交互连接，身份和目录符合文档。
- 三端提交关系清晰；任何工作树修改都已逐项记录且未被丢弃。
- 两端 `Agent/` 都恰好包含五份规定文档，逐文件 SHA-256 完全一致。
- 服务器授权目录边界、根盘风险和项目大盘状态已进入机器可读快照。
- GPU 枚举满足路径 A 的 8/8/8，或满足路径 B 的管理员白名单、排除结论和运行时映射一致；无解释的 8/7 不通过。
- 候选四卡 volatile 不可纠正 ECC 为 0，retired pages/row remap 没有 pending/failure，观察窗口无新增 ECC/Xid，且无活动计算进程。
- 历史 aggregate 不可纠正 ECC 非零的设备必须有管理员明确处置结论，不能仅凭“计数没有继续增长”自动进入候选四卡。
- 旧 `READY` 没有被当作本轮通过证据。
- 快照不含秘密、临时下载地址或其他用户信息。

## Gate 与后续依赖

- 远程访问、授权路径、仓库基线、用户修改和 `Agent/` 哈希全部通过后，形成 **G0-C 基线与安全子 gate**。
- GPU 枚举、管理员健康结论和 ECC 重验通过后，形成 **G0-G GPU 健康子 gate**。
- G0-C 与 G0-G 同时通过才形成 **G0 基线总 gate**。
- 远程访问或目录身份不通过时，S0.2 以后全部停止。
- G0-G 不通过时，可以继续 S0.2–S0.5 中明确不占用 GPU 的工作，但 S0.6 的 GPU 部分、S0.7、S0.10 的 GPU 实测和所有正式训练禁止执行。
- `Agent/` 哈希不一致时 G0-C 不通过；可以继续本地起草，但不能宣称任何阶段完成或进行最终同步。
