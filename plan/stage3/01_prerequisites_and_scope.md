# S3.1 前置 gate、范围与现状冻结

## 目的

在写真实模型 Stage 3 代码或启动服务器实验前，确认 Stage 0–2 的前置成果、服务器资源、资产身份和多端状态都达到可执行标准，并把本轮研究范围冻结。这个子任务防止用旧归档、未验收资产或故障硬件补齐一个表面完整但不可复核的结果。

## 当前依据

- 当前 `main` 没有现行源码、测试、配置或 Stage 1/2 正式结果。
- Stage 0–2 的详细计划正在当前工作树中并行起草，但 G0–G10、G1-EXIT、G2.7/G2.8 均没有现行通过报告。
- 本机有必须保留的用户修改，且本机/服务器 `Agent/worklogs.md` 哈希不一致。
- 服务器可见 7 张 A100，其中 GPU 0 有不可纠正 ECC 记录；正式实验不能开始。
- Pythia 14M 与 31M 资产目录存在，但 31M manifest 仍需修复和重验。
- Pile shard 0 与 idx 已验收；活动 shard 5 下载不在本阶段输入范围。
- 旧积分结果已经归档，但旧源码和 checkpoint 已清理。

## 前置条件

- 只使用 `Agent/remote_access.md` 定义的 SSH 别名和授权目录。
- 盘点阶段只读，不修改 GPU、驱动、下载任务、活动锁或旧归档。
- 所有当前修改先被完整识别，任何 Git 操作都遵守 `Agent/git.md`。

## 实施步骤

### 1. 建立 Stage 0–2 证据矩阵

1. 列出 Stage 0 的全部 gate 和对应报告路径。
2. 标记每个 Stage 0 gate 为“通过、失败、缺失或已漂移”。
3. 核对单卡、四卡、checkpoint 恢复和 provenance 是否有现行证据。
4. 列出 Stage 1 必须交付的参数映射、损失归约、梯度尺度和状态恢复测试。
5. 核对 Stage 1 的测试是否在现行提交和服务器锁定环境中零失败、零未解释 skip。
6. 列出 Stage 2 必须交付的主估计器选择、独立统计单元和偏差/方差报告。
7. 核对 Stage 2 是否已经明确局部 U-statistic 的目标与 AdamW 完整路径积分的边界。
8. 为每项证据记录提交、配置、数据/模型 revision、结果路径和 SHA-256。
9. 任何缺失项都登记为阻塞，不用旧归档或口头结论补位。
10. 将 Stage 1 的最终状态明确绑定到 `G1-EXIT`。
11. 将 Stage 2 的主估计器决策明确绑定到 `G2.7`。
12. 将 Stage 2 的交付同步明确绑定到 `G2.8`。

### 2. 复核仓库与多端状态

1. 记录本机当前分支、HEAD、工作树和用户已有修改。
2. 只读核对 GitHub 目标分支 HEAD。
3. 记录服务器分支、HEAD 和工作树。
4. 对比三端提交关系，确认没有分叉或服务器未提交修改。
5. 核对 `Agent/` 两端文件集合、大小和 SHA-256。
6. 先解决 `Agent/worklogs.md` 的内容差异，再宣称同步 gate 通过。
7. 为 Stage 3 决定独立开发分支和提交边界。
8. 将用户已有修改纳入审查计划，禁止还原、隐藏或选择性遗漏。

### 3. 复核硬件与运行窗口

1. 分别记录驱动层、NVML 和 PyTorch 的 GPU 枚举数量与映射。
2. 记录候选设备的不可纠正 ECC、显存占用和活动计算进程。
3. 要求 GPU 枚举异常和 ECC 异常先经过 Stage 0 规定的管理员处理或明确基线确认。
4. 从健康设备中选择一个逻辑候选单卡池，不在配置中固化物理编号。
5. 验证候选设备在正式运行前仍为空闲且健康。
6. 记录项目大盘空间、inode、读写权限和系统根盘余量。
7. 核对 Pile 下载和其他重 I/O 任务状态。
8. 为正确性运行与成本运行分别定义资源窗口；成本运行必须使用安静窗口。
9. 不通过暂停、删除或修改既有下载任务来制造安静窗口。

### 4. 复核模型与数据资产

1. 解析 Pythia 14M 的模型、config 和 tokenizer manifest。
2. 完全离线加载 14M 并核对参数量、dtype、参数名和 tokenizer 身份。
3. 修复并重新发布合法的 Pythia 31M manifest。
4. 完全离线加载 31M 并执行相同核验。
5. 核对 Pile shard 0 和 idx 的固定 revision、大小、SHA-256 与覆盖范围。
6. 冻结 Stage 3 所需的 update 与 probe 数据游标区间。
7. 证明这些区间只落在已经验收的 shard 0 中。
8. 证明 update、pilot probe、formal probe 和重放审计区间互不重叠。
9. 确认资产解析器会拒绝 `.part`、非法 JSON、未知 revision 和路径漂移。

### 5. 冻结阶段范围与实验身份

1. 将 14M 定义为主验证模型。
2. 将 31M 定义为跨规模确认模型。
3. 将 160M/410M 排除出高密度参考积分。
4. 将单卡 FP32 定义为真实模型数值验证主精度。
5. 将 FP64 定义为解析测试和选定真实状态的精度审计，而不是全量正式配置。
6. 将 full-update 端点线性路径定义为主路径。
7. 把 data-only 反事实路径标为有条件的次要诊断。
8. 为 pilot、formal-14m、formal-31m 和 replay 分配互不覆盖的实验命名空间。
9. 为每类运行固定结果目录、父子关系和重试规则。
10. 在工作日志中记录包含项、排除项、已知阻塞和责任边界。

## 产出

- Stage 0–2 证据矩阵；
- 本机/GitHub/服务器状态与 `Agent/` 哈希报告；
- GPU、存储、活动任务和运行窗口快照；
- Pythia 14M/31M 与 Pile 输入资产验收摘要；
- update/probe 数据区间分配表；
- Stage 3 范围、分支、实验 ID 和目录命名决策；
- 阻塞项、处理责任和复核日期清单。

## 核验标准

- Stage 0 的 G0–G10 必须全部通过；G3-0 不接受以范围缩减、人工 waiver 或“只换一张看似空闲的卡”替代这些前置 gate。
- 用户批准的范围缩减只能删除 Stage 3 的可选分析或收窄最终结论，不得豁免硬件健康、资产完整性、状态正确性、同步或其他 Stage 0 必需 gate。
- Stage 1 的 `G1-EXIT` 已证明参数映射、固定损失、状态恢复和数值不变量正确。
- Stage 2 的 `G2.7` 和 `G2.8` 已明确统计估计器结论与交付状态，且没有把一般 AdamW 路径积分声明为 U-statistic 的严格无偏目标。
- 三端提交关系清楚，用户修改未丢失，`Agent/` 五文件两端哈希一致。
- 正式候选 GPU 经过健康确认；异常设备不在候选池。
- 14M 与 31M 资产完全离线加载，所有 Stage 3 数据游标落入已验收范围。
- update 与所有 probe 区间在样本/序列级别不重叠。
- 活动 `.part`、锁和下载进程未被读取、移动、改名或干预。
- 运行范围和不在范围内的内容已写入工作日志。

## Gate 与后续依赖

- 本子任务通过形成 **G3-0 开工 gate**。
- G3-0 未通过时，可以继续 CPU 解析测试、接口设计和文档审查，但不能运行真实模型 Stage 3 pilot 或正式实验。
- GPU 异常未解决时，不得通过“改用另一张看似空闲的卡”绕过 Stage 0 硬件结论。
- 31M 未 ready 时，可开发 14M 管线，但不能完成 G3-6 正式结果完整性 gate。

## 失败与恢复

- 若前置报告缺失，回到对应 Stage 0/1/2 补齐，而不是在 Stage 3 内重复发明一套隐式验证。
- 若三端分叉或服务器有未知修改，停止同步并按 `Agent/sync.md` 处理。
- 若硬件健康状态恶化，保留已完成原子单元，撤销设备租约，待管理员处理后从新 session 继续。
- 若 Pile 活动下载影响成本测量，只推迟成本批次；不得终止或修改下载链路。
