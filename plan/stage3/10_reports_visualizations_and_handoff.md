# S3.10 报告、可视化、重放与 Stage 4 交接

## 目的

把 Stage 3 的代码、配置、端点、原始结果、统计表、可视化和方法决策整理成可重放、可追溯、符合存储边界的完整交付，并生成 Stage 4 能机器校验的推进 gate。

## 前置条件

- G3-7 通过，或已经形成明确的 Stage 3 失败结论。
- 原始结果、派生表和决策文件都有稳定 schema 与输入哈希。
- 所有大型产物保留在服务器 `DATA_ROOT`，本机和 Git 只接收适合版本控制的小型证据。

## 实施步骤

### 1. 整理服务器大型产物

1. 将端点原始状态放入按 run ID 隔离的 checkpoint/结果路径。
2. 将逐节点诊断放入对应原子单元目录。
3. 将参数级贡献按端点、probe、方法和参数注册表分片。
4. 将参考加密中间结果与规范参考分开标识。
5. 将 formal 原始结果与派生分析目录分开。
6. 将图表服务器构建产物放入 `reports`。
7. 为每个大型文件记录绝对路径、大小和 SHA-256。
8. 为目录级文件集合生成 manifest。
9. 不把模型、checkpoint、完整参数数组或缓存复制到 Git。
10. 不清理旧失败或诊断产物，除非另有精确清单和用户授权。

### 2. 整理 Git 小型产物

1. 纳入数学与实验合同。
2. 纳入配置 schema 和冻结的 resolved config 模板。
3. 纳入解析单元测试和小型 oracle 数据。
4. 纳入机器可读指标摘要。
5. 纳入 gate summary。
6. 纳入方法 recommendation JSON。
7. 纳入体积适合 Git 的图表源数据。
8. 纳入中文报告、图表和工作日志。
9. 纳入服务器大型产物 manifest 的小型副本或摘要。
10. 扫描秘密、临时 URL、设备连接细节和不应提交的大文件。

### 3. 生成规范化结果表

至少生成并冻结以下逻辑表：

1. 端点身份与训练诊断表。
2. Probe 身份、独立性和有效 token 表。
3. 参考加密收敛表。
4. 每方法、每状态、每 probe 的完备性表。
5. 逐参数向量误差摘要表。
6. 排序、top-q 和符号一致率表。
7. 层/模块总量、均值和质量分布表。
8. 数值误差与 probe 波动分解表。
9. 节点、wall-clock、显存和 I/O 成本表。
10. Gate observed/threshold/pass 表。
11. 失败、中断、重试和允许排除表。
12. 表之间的主键和父子关系说明。

### 4. 生成核心可视化

1. 生成“唯一梯度求值数—向量误差”的双对数收敛图。
2. 标注解析理论斜率和真实模型浮点平台期。
3. 生成各方法完备性残差的端点散点/箱线图。
4. 生成精度—wall-clock Pareto 图。
5. 生成节点数—Spearman/top-q overlap 曲线。
6. 生成 `state × method` 的 pass/fail 与误差热力图。
7. 生成路径 loss 随 \(\alpha\) 的代表曲线。
8. 在同图或配对图中生成方向导数随 \(\alpha\) 的曲线。
9. 生成数值误差与 probe 抽样波动的嵌套分解图。
10. 生成候选与参考的层/模块总量、均值和质量占比对照热图。
11. 生成误差对更新范数、曲率代理和位移缩放的关系图。
12. 生成 14M/31M 与 early/middle/late 的分面比较图。

### 5. 核验可视化

1. 为每张图记录源表路径和哈希。
2. 核对图中状态、模型、seed 和 probe 数与 manifest 一致。
3. 核对坐标轴单位和对数变换。
4. 对 near-zero 和 undefined 指标使用明确标记，不以零值填充。
5. 让 gate 阈值在图中可见。
6. 同时输出适合审阅的 PNG 和可缩放的 SVG。
7. 使用一致颜色映射和方法顺序。
8. 确认颜色和标记在灰度或色觉缺陷情况下仍可区分。
9. 从冻结源表全量重生成一次并比较输出摘要。

### 6. 编写 Stage 3 中文报告

报告至少包括：

1. 研究问题和不在范围内的内容。
2. 当前硬件、环境、模型、数据和提交身份。
3. 固定端点、线性路径和固定 probe 数学定义。
4. Full-update、data-only 和 local-U 的命名边界。
5. Pilot 与 formal 的分离和预注册过程。
6. 参考积分如何收敛和交叉确认。
7. 各候选的完备性和逐参数误差。
8. 排序、top-q、层/模块稳定性。
9. Probe 抽样波动与数值误差分解。
10. 14M/31M、阶段和更新尺度差异。
11. 成本与 Pareto 结果。
12. 默认方法、回退方法和适用边界。
13. 失败状态、限制和不能得出的结论。
14. 对旧归档结果的回归比较，但不混入正式统计。
15. Stage 4 的明确放行或阻断结论。

### 7. 生成机器可读 gate 摘要

1. 记录 Stage 0–2 前置证据摘要。
2. 记录 G3-0 至 G3-7 的状态。
3. 记录默认方法、节点、dtype 和路径类型。
4. 记录 formal 配置、阈值和参数注册表摘要。
5. 记录 14M 与 31M 覆盖数量。
6. 记录每项主指标的 observed、threshold 和 pass/fail。
7. 记录参考未解决单元数和 formal 缺失单元数。
8. 记录回退方法和触发条件。
9. 记录大型产物 manifest 和报告摘要。
10. 只有所有必需 gate 为 true 时设置 Stage 3 完成状态。

### 8. 执行三层重放

1. 在本机 CPU 重放全部解析求积测试。
2. 在服务器锁定环境重放全部测试且零未解释 skip。
3. 从一个冻结 14M 端点和 probe 无缓存重放参考与候选。
4. 比较重放参数贡献、指标和 gate 摘要。
5. 从 formal 原始结果重生成全部统计表。
6. 从统计表重生成全部图表和报告。
7. 比较重放前后文件集合、行数和稳定摘要。
8. 记录不可按位一致但在容差内的字段及原因。
9. 重放失败时不修补最终表，回到对应源阶段处理。

### 9. 形成 Stage 4 配置接口

1. 提供 `online_importance_method` 字段，默认引用 Stage 2 的 local-U 主指标。
2. 提供 `path_audit_rule` 字段，引用 Stage 3 默认求积规则。
3. 提供节点、权重和 dtype 字段。
4. 提供 full-update/data-only 路径类型字段。
5. 提供固定 probe 合同和 manifest 引用字段。
6. 提供审计频率和端点保存频率字段。
7. 提供自适应回退触发字段。
8. 提供超过适用范围时的停止或升级动作。
9. 提供 G3 gate summary 的路径和预期 SHA-256。
10. Stage 4 启动入口必须拒绝 gate 摘要缺失、失败或摘要不匹配。

### 10. 更新工作日志

1. 在目标和口径冻结时追加日志。
2. 在代码与解析测试完成时追加日志。
3. 在参考积分和 pilot 完成时追加日志。
4. 在每个 formal 批次完成、失败或中止时追加日志。
5. 在方法决策和 Stage 4 gate 形成时追加日志。
6. 记录提交、配置、seed、资产 revision、端点和结果路径。
7. 记录大型产物大小、SHA-256 和验收状态。
8. 记录硬件、I/O 背景、失败根因和安全恢复。
9. 不在日志中记录秘密、签名 URL、临时 CDN 地址或连接参数。

### 11. 审查 Git 工作树

1. 查看整个工作树，而不只查看 Stage 3 文件。
2. 审查用户已有 `docs/mathematics.md`、Stage 0/2 计划和其他并行修改。
3. 检查链接、Markdown、配置、测试和敏感信息。
4. 检查大型文件和禁止类型。
5. 运行适合本机的解析测试、静态检查和报告重建。
6. 运行服务器专属零 skip 测试和无缓存重放。
7. 按可独立复核边界形成提交。
8. 若用户修改尚不可验证或与当前实现冲突，保留并明确报告，不静默遗漏。

### 12. 完成多端同步

1. 推送经过审查的阶段提交到 GitHub 目标分支。
2. 推送前重新获取远端状态并检查非快进。
3. 只通过经过验证的 Git bundle 将服务器仓库快进到同一提交。
4. 不用 SCP 覆盖 Git 跟踪的代码文件。
5. 单独同步规定的五份 `Agent/` 文档。
6. 核对本机、GitHub 和服务器 HEAD。
7. 核对本机/服务器 `Agent/*.md` 文件集合和 SHA-256。
8. 核对服务器大型产物 manifest。
9. 精确清理本次临时 bundle。
10. 将最终同步状态写入工作日志。

## 产出

- 服务器大型端点、节点、参数贡献和结果 manifest；
- Git 中的小型合同、配置、测试、指标摘要、图表源数据和报告；
- 全套规范化结果表；
- PNG/SVG 核心可视化及源表映射；
- Stage 3 中文总报告；
- `stage3_gate_summary.json`；
- Stage 4 配置片段和启动前检查清单；
- CPU、服务器和无缓存端点三层重放报告；
- 持续更新的中文工作日志；
- 本机/GitHub/服务器与 `Agent/` 同步记录。

## 核验标准

- 每个报告结论都能追溯到 formal 端点、probe、原始贡献、配置和提交。
- 每张图能从已登记源表重新生成，样本数和阈值正确。
- 大型产物只在服务器大盘，Git 中没有模型、checkpoint、完整参数数组或缓存。
- 三层重放均通过，服务器专属测试没有未解释 skip。
- Stage 4 配置明确分开在线 local-U 与完整路径审计。
- Stage 4 启动检查能拒绝缺失或失败的 Stage 3 gate。
- 工作日志记录成功、失败、中断、修复和同步，不含秘密。
- 本机、GitHub 和服务器 Git 提交一致，`Agent/` 五文件哈希一致。

## Gate 与后续依赖

- 本子任务通过形成 **G3-8 交付 gate**。
- 只有 G3-0 至 G3-8 全部通过，才能宣称“第 3 阶段完成”并进入 Stage 4。
- 若 G3-7 形成的是失败结论，S3.10 仍应完整交付失败报告和重放证据，但不得把 G3-8 标记为 Stage 4 放行。
- Stage 4 必须读取机器可读 gate，而不是只依赖报告中的文字结论。

## 失败与恢复

- 若报告与源表不一致，修复生成器并从冻结源表重建，不手工编辑结果数字。
- 若 Git 工作树包含不可验证并行修改，暂停提交和同步，保留全部内容并向用户报告。
- 若远端出现非快进或服务器工作树不干净，停止同步并按 `Agent/git.md`、`sync.md` 处理。
- 若大型 manifest 不完整，不复制或删除产物，先补齐身份和哈希。
