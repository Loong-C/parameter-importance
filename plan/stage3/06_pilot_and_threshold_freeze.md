# S3.6 小规模 pilot 与正式阈值冻结

## 目的

用与正式数据完全分离的小规模 14M 实验校准数值底噪、参考节点上限、运行成本、指标稳定常数和正式资源预算，并在查看正式结果之前冻结所有选择规则。Pilot 用于设计，不用于证明方法通过。

## 前置条件

- G3-0 至 G3-4 通过。
- Pilot 专用训练区间、probe panel 和结果目录已经与 formal 区间分离。
- Pilot 配置中不存在根据运行结果自动放宽 gate 的逻辑。

## 实施步骤

### 1. 冻结 pilot 覆盖

1. 使用 Pythia 14M 和一条专用短程训练轨迹。
2. 固定一个 pilot 训练 seed。
3. 在训练前预先指定 early、middle、late 三个阶段边界。
4. 每个阶段预先选择至少两个端点位置。
5. 让端点覆盖较小和较大更新范数。
6. 让端点覆盖裁剪触发与未触发情形；若 pilot 中没有触发，记录事实而不伪造。
7. 为每个端点使用两个独立 pilot probe panel。
8. Pilot update 和 probe 数据不进入 formal。
9. 端点选择不能参考积分误差、排序或 gate 结果。

### 2. 运行端点与 probe 管线 pilot

1. 生成并发布 pilot pre/post 端点。
2. 对每个端点执行全新进程重放。
3. 验证 full-update 路径。
4. 若 data-only 分解已通过，附加验证 data-only 路径。
5. 对每个 probe 重复端点 loss 和梯度求值。
6. 验证不同 probe microbatch 切分的 token 加权一致性。
7. 验证分析前后状态不变。
8. 记录任何端点捕获、重放或确定性失败。

### 3. 运行参考加密 pilot

1. 对每个 pilot 单元运行预设 Gauss 加密序列。
2. 对相同单元运行复合 Simpson 加密序列。
3. 记录最早达到家族内收敛的级别。
4. 记录最早达到跨家族一致的级别。
5. 记录完备性残差进入平台的节点数。
6. 记录未收敛、近零损失变化和大抵消状态。
7. 对预选状态执行 FP64 审计。
8. 估算 formal 参考所需的最大节点上限。

### 4. 运行候选方法 pilot

1. 比较左端点。
2. 比较右端点。
3. 比较中点。
4. 比较单区间梯形。
5. 比较单区间 Simpson。
6. 比较预选的低阶复合梯形和复合 Simpson。
7. 比较低阶 Gauss-Legendre 候选。
8. 对每个方法计算完整指标字典。
9. 检查旧归档提示的“左端点层级失败”是否仍是风险，但不以复现成功与否作为当前 gate。
10. 记录每个方法的节点数、wall-clock 和峰值显存。

### 5. 校准稳定常数与 active-set

1. 汇总端点损失差的量级。
2. 汇总参考贡献 \(L_1\)、\(L_2\) 和最大绝对值量级。
3. 汇总未移动和近零贡献坐标比例。
4. 根据解析底噪设定完备性分母稳定常数。
5. 根据 FP32/FP64 差异设定 active 坐标阈值。
6. 验证 active-set 不会删除大量有意义贡献。
7. 验证小幅改变稳定常数不会反转方法选择。
8. 一旦 formal 开始，稳定常数和 active-set 规则锁定。

### 6. 冻结正式阈值

1. 将参考家族内加密阈值冻结。
2. 将跨家族参考阈值冻结。
3. 将参考完备性绝对和尺度化阈值冻结。
4. 将候选归一化 \(L_1/L_2/L_\infty\) 阈值冻结。
5. 将 active 参数 Spearman 阈值冻结。
6. 将 top-0.1%、top-1% 和 top-5% overlap/Jaccard 阈值冻结。
7. 将符号一致率阈值冻结。
8. 将层/模块总量、均值和质量分布阈值冻结。
9. 将每项指标的中位数、95 分位和最坏状态规则冻结。
10. 正式阈值不得比 S3.2 的原则上限更宽。
11. 把阈值写入版本化配置并计算 SHA-256。

### 7. 冻结正式实验矩阵

1. 冻结 14M 的训练 seed 数。
2. 冻结 14M 每个训练阶段的端点数。
3. 冻结 31M 的确认端点数。
4. 冻结每个端点的 probe panel 数。
5. 冻结候选方法和节点集合。
6. 冻结参考最高节点预算。
7. 冻结 FP64 审计状态选择规则。
8. 冻结反事实位移缩放是否进入次要分析。
9. 冻结允许排除的失败类型。
10. 冻结 formal 数据游标和结果目录。
11. 冻结 pilot 与 formal 不共享结果缓存的规则。

### 8. 冻结资源与时间预算

1. 估算一个端点/一个 probe 的端点加载时间。
2. 估算每个求积节点的平均前向/反向时间。
3. 估算参考加密最坏节点数。
4. 估算 14M formal 的 GPU 小时。
5. 估算 31M formal 的 GPU 小时。
6. 估算参数贡献、节点诊断和图表源数据体积。
7. 加入失败重试和无缓存重放余量。
8. 验证大盘仍满足预计新增量加安全余量。
9. 为成本测量预留安静窗口。
10. 超出预算时优先减少冗余 probe 或参考上限前必须重新论证；不能只删失败状态。

### 9. 处理 pilot 发现的问题

1. 将实现错误与研究现象分开记录。
2. 实现错误修复后使用新 pilot run ID 全量重跑受影响单元。
3. 参考未收敛时先诊断固定损失和精度。
4. Probe 波动过大时扩大 formal probe 或增加独立 panel。
5. 所有低节点候选都明显失败时，仍冻结包含复合/自适应候选的正式矩阵。
6. 不因 pilot 某方法表现好而删掉预先需要的对照方法。
7. 不把 pilot 数值并入 formal 统计。

### 10. 发布预注册包

1. 保存 formal resolved config 模板。
2. 保存阈值配置。
3. 保存状态选择算法和阶段边界。
4. 保存 update/probe 数据范围摘要。
5. 保存候选和参考节点清单。
6. 保存成本预算和停止规则。
7. 保存允许排除与失败分类规则。
8. 保存 pilot 报告及其输入/输出哈希。
9. 在工作日志中记录正式配置冻结时间和提交。
10. Formal 启动后任何合同变更都必须创建新版本并使旧 formal 批次失效。

## 建议的 formal 最小覆盖

Pilot 通过后，若资源估算没有要求扩大，正式矩阵至少采用：

- 14M：2 个训练 seed × 3 个阶段 × 每阶段 4 个端点，共 24 个端点；
- 31M：1 个训练 seed × 3 个阶段 × 每阶段 3 个端点，共 9 个端点；
- 每个端点至少 3 个相互独立的 formal probe panel；
- 每个模型至少包含低、中、高更新范数层，并记录裁剪状态；
- 每个主方法都在完全相同的端点和 probe 上配对比较。

如果 pilot 发现方差或阶段差异大，应在 formal 开始前扩大覆盖；不得在 formal 结果不理想后选择性补样本。

## 产出

- 6 个或更多 14M pilot 端点及独立 probe 结果；
- 参考节点上限与 FP32/FP64 pilot 报告；
- 稳定常数、active-set 和指标实现校准；
- 冻结的正式阈值配置；
- 冻结的 14M/31M formal 实验矩阵；
- GPU 时间、显存和存储预算；
- Pilot 问题、修复与重跑记录；
- 可审查的正式预注册包和摘要哈希。

## 核验标准

- Pilot 和 formal 的训练数据、probe、端点和结果目录完全分离。
- Pilot 覆盖 early/middle/late、不同更新范数和至少两个独立 probe。
- 所有正式指标、阈值、状态选择规则和允许排除条件在 formal 前冻结。
- 阈值不宽于 S3.2 的原则上限，并能由解析底噪支持。
- 参考节点预算能让 pilot 的每个主单元收敛。
- Formal 最小覆盖和资源预算可在当前大盘与健康单卡条件下完成。
- Pilot 修复没有覆盖失败证据，受影响结果使用新 ID 重跑。
- Pilot 数值不进入 formal 通过率。

## Gate 与后续依赖

- 本子任务通过形成 **G3-5 pilot 冻结 gate**。
- G3-5 通过后才能启动 S3.7 formal。
- Formal 启动后若更改目标损失、端点选择、probe 范围、参考规则、阈值或允许排除条件，必须废弃受影响 formal 批次并重新预注册。

## 失败与恢复

- 若 pilot 参考大量不收敛，返回 S3.2–S3.5 处理，不能带病进入 formal。
- 若资源预算超过可用窗口，优先优化流式执行和节点复用；任何缩减矩阵都要在 formal 前说明会失去的结论范围。
- 若只有 14M 资产可用，可以完成 pilot，但 G3-5 仍需包含明确的 31M 解阻计划。
